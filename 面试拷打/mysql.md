# MySQL 面试问题详细解析（结合项目）

## 基础概念与项目应用

### 1. 什么是 MySQL？它的主要存储引擎有哪些？项目中使用了哪种存储引擎？

**MySQL** 是一种关系型数据库管理系统，广泛应用于Web应用开发中。

**主要存储引擎**：
- **InnoDB**：默认存储引擎，支持事务、行级锁、外键约束，适合高并发场景。
- **MyISAM**：不支持事务，适合读多写少的场景，如日志记录。
- **Memory**：内存存储，速度快但数据不持久。
- **Archive**：适合存储归档数据，压缩率高。

**项目中使用的存储引擎**：
项目默认使用 **InnoDB** 存储引擎，因为：
- 支持事务，确保数据一致性（如弹幕保存和视频统计更新的原子性）。
- 支持行级锁，提高并发性能（如多个用户同时发送弹幕的场景）。
- 支持外键约束，保证数据完整性。

### 2. 项目中 MySQL 的配置是如何管理的？核心配置参数有哪些？

**配置管理**：
在 `application.yml` 文件中配置 MySQL 连接信息：
```yaml
spring:
  datasource:
    driver-class-name: com.mysql.cj.jdbc.Driver
    url: jdbc:mysql://120.53.45.251:13306/tianmu?useUnicode=true&characterEncoding=UTF-8&serverTimezone=Asia/Shanghai
    username: root
    password: gK3T9n%q2M@j7Z9
```

**核心配置参数**：
- `driver-class-name`：数据库驱动类，使用 `com.mysql.cj.jdbc.Driver`（MySQL 8.0+ 推荐）。
- `url`：数据库连接地址，包含主机、端口、数据库名和连接参数。
  - `useUnicode=true`：使用 Unicode 字符集。
  - `characterEncoding=UTF-8`：字符编码为 UTF-8，支持中文。
  - `serverTimezone=Asia/Shanghai`：设置时区为上海，避免时间戳问题。
- `username` 和 `password`：数据库登录凭证。

### 5. 项目中如何处理数据的逻辑删除？

**实现方式**：
使用 MyBatis-Plus 的逻辑删除功能，在 `application.yml` 中配置：
```yaml
mybatis-plus:
  global-config:
    db-config:
      logic-delete-field: isDelete # 全局逻辑删除的实体字段名
      logic-delete-value: 1 # 逻辑已删除值（默认为 1）
      logic-not-delete-value: 0 # 逻辑未删除值（默认为 0）
```

**应用场景**：
- 当删除弹幕时，不是物理删除，而是将 `isDelete` 字段设置为 1。
- 查询时，MyBatis-Plus 会自动过滤 `isDelete=0` 的记录。

**优势**：
- 保留数据历史，便于审计和恢复。
- 避免物理删除带来的数据丢失风险。
- 简化业务逻辑，无需手动处理删除状态。

### 6. 什么是数据库索引？项目中哪些表使用了索引？

### 索引设计详解
1. 索引类型与选择

- 主键索引 ：每个表的主键自动创建，使用 B+ 树结构，查询效率最高。
- 唯一索引 ：确保字段值唯一，同时提高查询效率，如 User 表的 username 和 email 字段。
- 普通索引 ：加速查询，但允许重复值，如 Video 表的 user_id 字段。
- 复合索引 ：多个字段组合的索引，如 Like 表的 user_id 和 video_id 组合索引。
- 前缀索引 ：对长字符串字段创建前缀索引，平衡索引大小和查询效率。
2. 项目中的索引设计

## 事务与并发

### 11. 什么是数据库事务？项目中哪些操作使用了事务？

**数据库事务**：
是一组原子性的 SQL 操作，要么全部执行成功，要么全部失败回滚。

**项目中使用事务的操作**：
- **保存弹幕**：在 `BulletServiceImpl.saveBulletToMySQL()` 方法中，使用 `@Transactional` 注解确保弹幕保存和视频统计更新的原子性：
  ```java
  @Override
  @Transactional(rollbackFor = Exception.class)
  public void saveBulletToMySQL(SendBulletRequest sendBulletRequest) {
      // 校验视频和用户
      // 初始化视频统计
      // 更新弹幕计数
      // 保存弹幕
  }
  ```
- **删除弹幕**：在 `BulletServiceImpl.deleteVideoBullet()` 方法中，使用事务确保弹幕删除和视频统计更新的原子性。
- **其他操作**：如用户注册、视频发布等涉及多表操作的场景。

### 12. MySQL 的事务隔离级别有哪些？项目中使用了哪种隔离级别？

**MySQL 的事务隔离级别**：
- **READ UNCOMMITTED**：读未提交，可能读取到其他事务未提交的数据（脏读）。
- **READ COMMITTED**：读已提交，只能读取到其他事务已提交的数据，避免脏读，但可能出现不可重复读。
- **REPEATABLE READ**：可重复读，同一事务中多次读取同一数据结果一致，避免脏读和不可重复读，但可能出现幻读。
- **SERIALIZABLE**：串行化，最高隔离级别，避免所有并发问题，但性能最差。

**项目中使用的隔离级别**：
项目使用 MySQL 的默认隔离级别 **REPEATABLE READ**，因为：
- 避免了脏读和不可重复读，保证数据一致性。
- 性能比 SERIALIZABLE 好，适合高并发场景。
- 对于弹幕系统，幻读的影响较小。

### 13. 项目中如何处理并发操作？比如多个用户同时发送弹幕的场景。

**并发处理措施**：
- **事务**：使用 `@Transactional` 确保操作的原子性。
- **乐观锁**：通过唯一 ID（如 `bulletId`）避免重复插入。
- **原子操作**：使用 `setSql("bullet_count = bullet_count + 1")` 进行原子更新，避免并发更新问题：
  ```java
  boolean updated = videoStatsService.lambdaUpdate().setSql("bullet_count = bullet_count + 1").eq(VideoStats::getVideoId, videoId).update();
  ```
- **Redis 缓存**：使用 Redis 缓存热点数据，减少数据库压力。

**具体实现**：
- 当多个用户同时发送弹幕时，通过 `bulletId` 的唯一性确保不会重复插入。
- 使用原子操作更新视频统计数据，避免并发更新导致的数据不一致。

### 14. 什么是死锁？项目中如何避免死锁？

**死锁**：
是指两个或多个事务相互等待对方释放资源，导致所有事务都无法继续执行的情况。

**死锁产生的条件**：
- 互斥条件：资源不能被多个事务同时使用。
- 请求与保持条件：事务已经持有一些资源，同时又请求其他资源。
- 不剥夺条件：资源只能由持有它的事务主动释放。
- 循环等待条件：多个事务形成循环等待资源的关系。

**项目中避免死锁的措施**：
- **统一操作顺序**：所有事务按照相同的顺序操作表，避免循环等待。
- **减少事务范围**：尽量缩短事务执行时间，减少锁的持有时间。
- **使用索引**：通过索引减少锁的范围，避免全表扫描。
- **避免长事务**：将复杂事务拆分为多个短事务。
- **使用乐观锁**：对于并发冲突较少的场景，使用乐观锁替代悲观锁。

**项目中的应用**：
- 在 `BulletServiceImpl` 中，使用乐观锁和原子操作，减少死锁的可能性。
- 事务操作顺序一致，先校验后更新，避免循环等待。

### 15. 项目中是否使用了乐观锁或悲观锁？如何实现？

**项目中使用了乐观锁**。

**实现方式**：
- **唯一 ID**：通过 `bulletId` 的唯一性确保弹幕不会重复插入：
  ```java
  // 生成唯一 bulletId
  bullet.setBulletId(System.currentTimeMillis() + ThreadLocalRandom.current().nextLong(1000, 9999));
  ```
- **幂等性检查**：在消费者端检查弹幕是否已存在：
  ```java
  if (bulletService.bulletExists(bulletId)) {
      log.info("弹幕已存在，跳过保存，bulletId: {}", bulletId);
      return;
  }
  ```
- **原子操作**：使用 `setSql` 进行原子更新，避免并发更新问题：
  ```java
  videoStatsService.lambdaUpdate().setSql("bullet_count = bullet_count + 1").eq(VideoStats::getVideoId, videoId).update();
  ```

**优势**：
- 避免了悲观锁的性能开销，适合高并发场景。
- 实现简单，无需额外的锁机制。

## 架构设计与未来规划

### 25. MySQL 在整个项目架构中的位置是什么？与其他组件（如 Redis、RocketMQ）的关系如何？

**MySQL 在架构中的位置**：
- **核心存储**：存储所有业务数据，如用户信息、视频数据、弹幕数据等。
- **数据持久化**：确保数据的持久化存储，即使系统重启也不会丢失数据。
- **事务支持**：提供事务处理能力，确保数据一致性。

**与其他组件的关系**：
- **与 Redis 的关系**：
  - Redis 作为缓存，存储热点数据（如弹幕列表），减少数据库压力。
  - MySQL 作为持久化存储，Redis 中的数据最终会同步到 MySQL。
- **与 RocketMQ 的关系**：
  - RocketMQ 作为消息中间件，异步处理弹幕消息。
  - 消费者从 RocketMQ 接收消息后，将数据存储到 MySQL。
- **与 WebSocket 的关系**：
  - WebSocket 服务器接收前端消息，通过 RocketMQ 异步存储到 MySQL。
  - WebSocket 服务器从 Redis 或 MySQL 获取数据，实时推送给前端。

**架构流程图**：
```
前端 → WebSocket 服务器 → RocketMQ → 消费者 → MySQL
     ↓                ↓
     Redis            Redis (缓存)
```

### 26. 对比其他数据库（如 PostgreSQL、MongoDB），MySQL 的优势和劣势是什么？

**MySQL 的优势**：
- **成熟稳定**：广泛应用于生产环境，社区活跃。
- **性能优异**：针对 OLTP 场景优化，适合高并发读写。
- **易于使用**：配置简单，学习曲线平缓。
- **生态丰富**：有大量工具和第三方集成。
- **成本低**：开源免费，部署成本低。

**MySQL 的劣势**：
- **功能相对简单**：高级特性（如复杂查询、JSON 支持）不如 PostgreSQL。
- **扩展性有限**：单机性能有限，需要通过分库分表等方式扩展。
- **数据一致性**：在高并发场景下，可能出现数据一致性问题。
- **事务支持**：虽然支持事务，但在高并发下性能可能下降。

**PostgreSQL 的优势**：
- **功能丰富**：支持复杂查询、JSON、GIS 等高级特性。
- **数据一致性**：事务支持更完善，适合复杂业务场景。
- **扩展性强**：支持自定义类型和函数。

**MongoDB 的优势**：
- **灵活的数据模型**：适合存储非结构化数据。
- **水平扩展**：支持分片，易于水平扩展。
- **查询灵活**：支持丰富的查询操作。
