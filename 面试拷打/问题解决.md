# 线上故障处理实际案例：WebSocket 连接异常导致弹幕发送失败

## 故障背景

**时间**：2025年11月15日 20:30（业务高峰期）  
**系统**：视频弹幕平台（4核4G服务器，Netty WebSocket服务）  
**现象**：用户反馈弹幕发送后不显示，部分用户无法连接WebSocket，后台监控显示消息队列堆积。

## 故障现象

1. **用户侧**：
   - 弹幕发送后无响应，界面显示"发送失败"
   - 部分用户刷新页面后无法重新连接WebSocket
   - 热门视频弹幕数量骤减（从1000条/分钟降至100条/分钟）

2. **监控侧**：
   - WebSocket连接数从15,000降至8,000
   - RocketMQ消息堆积：积压5,000+条弹幕消息
   - 服务器CPU使用率：85-90%（正常30-50%）
   - 内存使用率：85%（正常60-70%）

## 故障排查过程

### 1. 紧急响应（5分钟内）

- **Step 1：查看系统监控**
  - 发现WebSocket连接数急剧下降，CPU使用率飙升
  - 检查Netty服务日志：大量"ChannelClosed"异常
  - 检查RocketMQ监控：消费端处理速度从8,000条/秒降至1,000条/秒

- **Step 2：定位问题源头**
  - 查看Netty服务线程状态：工作线程（4个）全部处于"RUNNABLE"状态
  - 检查内存使用：老年代内存使用率90%，触发频繁Full GC
  - 分析GC日志：每30秒一次Full GC，每次持续1-2秒

### 2. 根因分析（10分钟内）

- **现象**：Netty工作线程阻塞 + 内存泄漏
- **排查**：
  1. 检查WebSocketHandler代码：发现`broadcastMessage`方法中存在同步调用Redis操作
  2. 查看Redis监控：Redis响应时间从1ms增至200ms
  3. 分析网络：Redis服务器与应用服务器之间网络延迟从10ms增至100ms
  4. 定位代码：
     ```java
     // 问题代码
     private void broadcastMessage(String videoId, String message) {
         // 同步调用Redis，阻塞Netty工作线程
         redisTemplate.opsForZSet().add("bullet:" + videoId, message, System.currentTimeMillis());
         ChannelGroup group = videoMap.get(videoId);
         if (group != null) {
             group.writeAndFlush(new TextWebSocketFrame(message));
         }
     }
     ```

- **根因**：
  1. 网络波动导致Redis响应延迟
  2. 同步Redis操作阻塞Netty工作线程
  3. 线程阻塞导致WebSocket连接无法处理，连接超时关闭
  4. 连接关闭后，堆积的消息无法及时消费，导致RocketMQ消息积压

## 故障解决方案

### 1. 临时缓解（15分钟内）

- **Step 1：隔离故障**
  - 重启Netty服务，释放阻塞线程
  - 临时将Redis操作改为异步：
    ```java
    // 临时修复
    private void broadcastMessage(String videoId, String message) {
        // 异步调用Redis，避免阻塞线程
        executorService.submit(() -> {
            redisTemplate.opsForZSet().add("bullet:" + videoId, message, System.currentTimeMillis());
        });
        ChannelGroup group = videoMap.get(videoId);
        if (group != null) {
            group.writeAndFlush(new TextWebSocketFrame(message));
        }
    }
    ```

- **Step 2：恢复服务**
  - 重启RocketMQ消费者，加速消息消费
  - 监控WebSocket连接数：30分钟内恢复至15,000
  - 监控消息堆积：1小时内完全消化5,000+条积压消息

### 2. 根本修复（24小时内）

- **Step 1：架构优化**
  - **异步化改造**：
    - 将所有Redis操作改为异步，使用CompletableFuture
    - 引入消息队列处理Redis写入，避免直接阻塞
  - **网络优化**：
    - 迁移Redis至与应用服务器同可用区，网络延迟降至1ms
    - 配置Redis连接池，提高连接复用率

- **Step 2：代码重构**
  ```java
  // 优化后代码
  private void broadcastMessage(String videoId, String message) {
      // 1. 优先广播消息（保证实时性）
      ChannelGroup group = videoMap.get(videoId);
      if (group != null) {
          group.writeAndFlush(new TextWebSocketFrame(message));
      }
      
      // 2. 异步写入Redis（保证可靠性）
      CompletableFuture.runAsync(() -> {
          try {
              redisTemplate.opsForZSet().add("bullet:" + videoId, message, System.currentTimeMillis());
          } catch (Exception e) {
              log.error("Redis write failed", e);
              // 降级：写入本地缓存，后续同步
          }
      }, executorService);
  }
  ```

- **Step 3：监控增强**
  - 增加Netty线程状态监控：线程阻塞时间超过100ms告警
  - 增加Redis响应时间监控：超过50ms告警
  - 增加WebSocket连接异常率监控：超过5%告警

## 故障影响与处理结果

- **影响范围**：约20%用户（3,000-4,000人）在30分钟内无法正常发送弹幕
- **恢复时间**：从发现故障到完全恢复共60分钟
- **长期效果**：
  - WebSocket连接稳定性提升：连接失败率从2%降至0.1%
  - 系统吞吐量提升：弹幕处理能力从8,000条/秒提升至12,000条/秒
  - 故障预警能力：类似问题可提前10分钟发现并处理

## 经验教训与改进措施

1. **技术层面**：
   - **异步化处理**：IO密集型操作（如Redis、数据库）必须异步化，避免阻塞核心线程
   - **网络依赖**：关键依赖服务（如Redis、MQ）应部署在同可用区，减少网络延迟
   - **监控覆盖**：增加线程状态、网络延迟等细粒度监控，提前发现潜在问题

2. **流程层面**：
   - **应急预案**：制定WebSocket服务故障应急预案，明确处理步骤
   - **代码审查**：加强对核心服务代码的审查，重点关注同步操作对性能的影响
   - **故障演练**：定期进行故障演练，提高团队应对能力

3. **架构层面**：
   - **服务降级**：设计降级策略，当Redis不可用时，使用本地缓存或消息队列保证核心功能
   - **弹性扩容**：WebSocket服务支持自动扩缩容，应对突发流量
   - **多活部署**：考虑多可用区部署，提高服务可用性
